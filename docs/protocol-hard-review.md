# Protocol Hard Review — v1.0 Readiness Audit

> **Rule**: This document identifies, prioritizes, and recommends. It does NOT implement fixes.
>
> Audit date: 2026-05-29
> Auditor: self (dev-protocol dogfood)
> Scope: commands, state model, skills, docs, workflow friction

---

## 1. What Works Well

These are the architectural strengths that should be preserved in v1.0.

| Strength | Evidence |
|---|---|
| **State-over-history principle** | Fresh session recovery works. `/dev-status` reconstructs context without chat history. |
| **Git reality priority hierarchy** | Consistently applied across all skills. Prevents stale state from overriding observable reality. |
| **Auto-execution criteria** | Reduces `/dev-scope` → `/goal` duplication for simple work. Validated by case-27/28/29. |
| **Continue loop with tolerant parsing** | Successfully executed 3 loops during dogfood. Scope derivation was accurate. |
| **Semantic validation layer** | Reduces false negatives from rigid string matching. Tested via case-37~40. |
| **Protocol / runtime separation** | Core logic lives in `skills/` and `.agents/`. `.claude/` is optional adapter only. |
| **Runtime-agnostic design** | No runtime-specific logic in protocol core. Compatible with Claude Code, Cursor, Copilot, manual. |
| **No-op save support** | Verification loops are first-class workflow outcomes. Validated by case-08/25. |
| **Static test suite** | `run-tests.ps1` provides deterministic keyword validation for all critical paths. |

## 2. Current Friction

Real friction points observed during dogfood and code audit. Each includes severity and reproduction context.

### F1. Path mismatch between `generate plan` and `continue loop` (HIGH)

`generate plan` writes to `docs/next-phase-plan.md`  
`continue loop` reads from `.agents/dev-protocol/next-phase-plan.md`

This means a plan generated by `generate plan` is invisible to `continue loop` unless manually moved or copied.

**Repro**: Run `generate plan`, then `continue loop`. Preconditions fail because plan is in wrong directory.

**Fix direction**: Unify to one path. Prefer `.agents/dev-protocol/next-phase-plan.md` since it is protocol state.

---

### F2. `goal-output` generation requires manual script execution (HIGH)

`project-rules.md` mandates running `scripts/fix-goal-output.ps1` after every goal to correct `changed_files`. The LLM cannot reliably generate accurate file lists.

**Repro**: Complete any `/goal`. Observe that `changed_files` in `goal-output.md` is often wrong until script runs.

**Impact**: This is a manual step the protocol cannot automate. It breaks the autonomous workflow promise.

**Fix direction**: Make `changed_files` generation deterministic without a post-hoc script. Options:
- Omit `changed_files` entirely; derive from git during `/dev-save`
- Make `/dev-save` auto-populate `changed_files` from `git diff-tree`

---

### F3. Drift detection fails on first run (MEDIUM)

`checkpoint.last_commit` is empty after `/dev-init`. First `/dev-status` or `continue loop` skips diff-based drift comparison. The user must run `/dev-save` before drift detection works.

**Repro**: `/dev-init` → make source commits → `/dev-status`. No drift detected because no baseline exists.

**Fix direction**: Initialize `checkpoint.last_commit` to current HEAD during `/dev-init` (with a flag noting it is an onboarding baseline, not a save).

---

### F4. Command overlap: `/dev-scope`, `/goal`, `continue loop` blur boundaries (MEDIUM)

| Command | Produces scope? | Executes? | Reads plan? |
|---|---|---|---|
| `/dev-scope` | Yes | Sometimes (auto-execution) | No |
| `/goal` | No | Yes | No |
| `continue loop` | Yes | Sometimes (auto-execution) | Yes |

`/dev-scope` auto-execution makes it overlap with `/goal`. `continue loop` derives scope, making it overlap with `/dev-scope`. The user cannot predict which command will execute vs. produce a document.

**Repro**: `/dev-scope "fix typo"` auto-executes. `/dev-scope "refactor auth"` produces scope. `continue loop` on a simple plan auto-executes. Same user intent, three different behaviors.

**Fix direction**: Consolidate execution authority. Either:
- `/dev-scope` NEVER executes; always produces scope
- `continue loop` is the ONLY auto-execution path
- `/goal` is the ONLY manual execution path

---

### F5. Semantic classification relies on LLM judgment (MEDIUM)

Semantic drift classification and equivalence rules require the LLM to interpret commit intent. This is inherently inconsistent.

Examples of ambiguity:
- `docs(protocol): harden contracts` — is this documentation-only (low drift) or source-impacting (high drift)?
- `refactor(tests): extract helpers` — is this test-only (low) or architectural (high)?
- "started work on" vs "completed" — LLMs frequently misclassify these

**Repro**: Review any commit history. Ask two different LLM sessions to classify drift. Results diverge.

**Fix direction**: Reduce semantic surface area. Make classification deterministic:
- File-extension-based rules (`.md` = low, `.py` with test prefix = low)
- Commit-prefix-only rules (strict, no semantic interpretation)

---

### F6. Legacy alias skills carry stale complexity (MEDIUM)

`dev-checkpoint/PROMPT.md` still claims "NEVER stage files, NEVER auto-commit" — contradicting v2 `/dev-save`.  
`dev-resume/PROMPT.md` uses old drift terms "none/minor/major".  
`dev-bootstrap/PROMPT.md` references legacy `.agent/` path.

These are deprecated aliases, but their PROMPT.md files are still in the repo and could be loaded accidentally.

**Fix direction**: Strip alias skills to minimal redirect stubs. Remove all PROMPT.md files from alias skills. Keep only SKILL.md with "Redirect to `/dev-xxx`".

---

### F7. `project-rules.md` contains stale and false information (LOW)

- "No git history on master branch yet" — false
- "/dev-save ... no git operations" — false, it auto-commits
- "Three state files required" — actually 4 if you count `goal-output.md` and `next-phase-plan.md`
- `current-focus.md` prevention rule exists but `current-focus.md` is still referenced in v1 skills

**Impact**: Stale rules reduce trust in the state system.

---

### F8. Test matrix is out of sync with reality (MEDIUM)

`docs/test-matrix.md` references case IDs that do not exist:
- `case-11-continue-loop` — actual test is `case-30`
- `case-11-aborted-goal` — no test directory exists
- `case-10-handoff-mismatch` — no test directory exists

`README.md` test table only shows cases up to `case-26`, missing `case-27~40`.

**Fix direction**: Sync test-matrix.md with actual `tests/` directory. Update README.md test table.

---

### F9. `/goal` has no skill definition (LOW)

`/goal` is documented in `command-contracts.md` but has no `skills/goal/` directory, no PROMPT.md, no SKILL.md. It is the only canonical command without a skill definition.

**Impact**: Inconsistent skill structure. `/goal` behavior is only defined in contracts, not in executable prompts.

**Fix direction**: Create `skills/goal/PROMPT.md` and `SKILL.md`, or remove `/goal` as a standalone command and merge into `/dev-scope` auto-execution.

---

### F10. Two `docs` directories create ownership ambiguity (LOW)

`docs/` contains public documentation and design documents.  
`.agents/dev-protocol/docs/` contains deferred improvements and runtime docs.

Neither location has a clear ownership boundary. `generate plan` reads from `docs/v2-redesign-roadmap.md` but writes plan to `docs/next-phase-plan.md` — yet `continue loop` expects it in `.agents/dev-protocol/`.

**Fix direction**: Consolidate or document ownership. Public docs → `docs/`. Runtime docs → `.agents/dev-protocol/docs/`. Never mix.

## 3. Protocol Debt

Architectural debt that will compound if not addressed before v1.0.

### D1. State file duplication

| Information | workflow-state.yml | handoff.md | goal-output.md | next-phase-plan.md |
|---|---|---|---|---|
| Focus | Yes | Yes | No | No |
| Phase | Yes | Implied | No | No |
| Completed work | Yes | Yes | Yes | No |
| Blockers | Yes | Yes | No | No |
| Next actions | Yes | Yes | No | Implied |
| Validation results | No | No | Yes | Yes |

Four files track overlapping context. A fresh session must read ALL of them to reconstruct state. This increases failure surface area.

**Debt**: High maintenance burden, risk of contradictions between files.

**Recommended fix**: Merge `goal-output.md` into `handoff.md` (Completed Since Last Save). Merge `next-phase-plan.md` into `workflow-state.yml` as a `plan:` section.

---

### D2. `dev-status` SKILL.md is out of sync with PROMPT.md

`dev-status/SKILL.md`:
- Missing "Semantic drift classification" section
- Missing "Semantic theme inference" section
- Has duplicated section numbering ("5. Output Summary" appears twice)
- Uses "low/medium/high" in one table but PROMPT.md uses "none/low/high"

**Debt**: SKILL.md and PROMPT.md diverge. Runtime behavior depends on which file the agent loads.

---

### D3. `command-contracts.md` continue-loop section omits semantic validation

The contracts document describes the execution sequence as:

```
→ 5. derive scope
→ 6. evaluate auto-execution criteria
→ 7. if auto-execute: execute
```

Missing STEP 4.5 (Semantic Validation Equivalence) that exists in `continue-loop/PROMPT.md`.

**Debt**: Contract document is stale relative to implementation.

---

### D4. Incident logging is defined only in deprecated skills

Incident logging (`incidents.md`) is documented in `dev-checkpoint` and `dev-resume` — both deprecated. No v2 skill mentions incidents.

**Debt**: Either migrate incident logging to v2 or remove it entirely.

---

### D5. `goal-output` dual format (JSON + Markdown)

`case-06` tests validate both JSON and Markdown fallback. This doubles test complexity and skill maintenance.

**Debt**: Every skill that produces goal-output must support both formats.

## 4. Simplification Opportunities

Opportunities to reduce surface area without losing capability.

### S1. Remove `/goal` as standalone command (HIGH IMPACT)

`/goal` is just "execute scoped work." `/dev-scope` already auto-executes simple scopes. `continue loop` already auto-executes planned loops.

**Proposed change**:
- `/dev-scope` ALWAYS produces a scope document (never auto-executes)
- If scope is simple, the user runs `/goal` explicitly
- `continue loop` auto-executes simple planned loops
- Complex work always requires explicit confirmation

**Benefit**: Clear command boundaries. No surprise execution.

---

### S2. Remove `goal-output.json` (MEDIUM IMPACT)

Keep only `goal-output.md`. Markdown is human-readable, editable, and sufficient for test validation.

**Benefit**: Halves test complexity. Removes JSON schema maintenance.

---

### S3. Strip v1 alias skills to stubs (MEDIUM IMPACT)

Remove all PROMPT.md files from alias skills. Keep only SKILL.md with:

```markdown
# /dev-bootstrap

**DEPRECATED**. Use `/dev-init` instead.

Redirect: /dev-init
```

**Benefit**: Eliminates stale content. Reduces repo size. Prevents accidental loading of outdated prompts.

---

### S4. Consolidate documentation directories (MEDIUM IMPACT)

Move `.agents/dev-protocol/docs/deferred-improvements.md` to `docs/deferred-improvements.md`.  
Keep `.agents/dev-protocol/docs/` for runtime-generated docs only.

**Benefit**: Single source of truth for human-readable docs.

---

### S5. Merge `next-phase-plan.md` into `workflow-state.yml` (LOW IMPACT)

Add a `plan:` section to `workflow-state.yml`:

```yaml
plan:
  loops:
    - name: "Update command contracts"
      status: pending
      files: [docs/command-contracts.md]
      validation: ["case-30 PASS"]
```

**Benefit**: One file to read. One file to update. No path mismatches.

## 6. v1.0 Exit Criteria

dev-protocol is stable enough for long-term self-hosted use when ALL of the following are true:

### 6.1 Command Surface

- [ ] `/dev-init`, `/dev-scope`, `/dev-save`, `/dev-status` behavior is frozen
- [ ] `generate plan` and `continue loop` paths are unified (no path mismatches)
- [ ] `/goal` either has a skill definition or is merged into `/dev-scope`
- [ ] Auto-execution behavior is deterministic and documented
- [ ] No command overlaps that surprise the user

### 6.2 State Model

- [ ] `workflow-state.yml`, `handoff.md`, `project-rules.md` are the ONLY required state files
- [ ] `goal-output.md` and `next-phase-plan.md` are optional, well-located, and non-duplicative
- [ ] No stale information in `project-rules.md`
- [ ] State file templates are validated on creation

### 6.3 Skills

- [ ] All canonical skills (`dev-init`, `dev-scope`, `dev-save`, `dev-status`, `continue-loop`, `generate-plan`) have consistent structure
- [ ] SKILL.md and PROMPT.md are synchronized for each skill
- [ ] v1 alias skills are stripped to minimal redirects
- [ ] `/goal` has a skill definition if it remains a standalone command

### 6.4 Documentation

- [ ] `README.md` test table matches actual test cases
- [ ] `docs/test-matrix.md` case IDs match `tests/` directory names
- [ ] `docs/command-contracts.md` reflects current implementation (including semantic layer)
- [ ] `docs/v2-redesign-roadmap.md` is accurate (no drift from reality)
- [ ] Public docs and runtime docs have clear ownership boundaries

### 6.5 Testing

- [ ] All active tests (case-05 through case-40) pass
- [ ] Test matrix has no orphaned or missing cases
- [ ] At least one regression test validates each canonical command
- [ ] Static validation covers all skill PROMPT.md and SKILL.md files

### 6.6 Validation

- [ ] Real-project validation executed on at least 2 external projects
- [ ] No manual script execution required during normal workflow
- [ ] Drift detection works from first use (no empty-baseline gap)
- [ ] Semantic layer validated with real commit patterns (not just synthetic tests)

### 6.7 Stability

- [ ] No HIGH severity friction findings remain open
- [ ] No MEDIUM severity findings older than 2 iterations remain open
- [ ] Commit convention is stable (no new `chore()` prefixes added)
- [ ] State file format has not changed for 3+ iterations

1. **Fix F1** (path mismatch) — unify `next-phase-plan.md` location
2. **Fix F6** (alias skill stale content) — strip to stubs
3. **Fix F7** (stale project-rules.md) — audit and update
4. **Fix F8** (test matrix sync) — align case IDs with directories

### Short-term (next 2 iterations)

6. **Address D2** (SKILL.md sync) — audit all SKILL.md vs PROMPT.md pairs
7. **Address F2** (goal-output script) — make changed_files deterministic without script

9. **S1** (remove /goal or give it a skill definition)
10. **S2** (remove goal-output.json)
11. **S4** (consolidate docs directories)
12. **S5** (merge plan into workflow-state)

- Auto-repair (too complex, risks unintended changes)
- Advanced hooks (runtime-specific, not protocol core)
- Cross-runtime hook lifecycle validation (deferred to post-v1.0)
- Constants audit automation (nice-to-have, not critical)

## 8. Audit Summary

| Category | Count | Notes |
|---|---|---|
| Friction findings | 10 | 1 HIGH, 5 MEDIUM, 4 LOW |
| Simplification candidates | 5 | 2 HIGH impact, 3 MEDIUM impact |
| Recommended removals | 6 | All low risk |
| Protocol debt items | 5 | All compound if not fixed |
| v1.0 exit criteria | 7 sections, 29 checkboxes | Must all be true for v1.0 |

**Overall assessment**: dev-protocol v2 is functionally solid but has accumulated organizational debt. The semantic layer and continue-loop features are strong additions. Before v1.0, the priority is **consolidation and deduplication** — not new features. The path mismatch (F1), stale alias skills (F6), and state model duplication (D1) are the highest-value fixes.

**Verdict**: v1.0 is achievable in 2–3 focused iterations if scope is strictly limited to exit criteria fulfillment.
